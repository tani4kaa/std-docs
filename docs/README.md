# Документація: Web-застосунок для управління замовленнями

Програмний продукт призначений для автоматизації життєвого циклу замовлень у сфері роздрібної торгівлі. Система забезпечує єдиний простір для взаємодії клієнтів та адміністраторів магазину.

## Цільова аудиторія документації
* **Розробники (Backend/Frontend):** потребують технічних специфікацій та опису API.
* **QA-інженери:** використовують документацію для написання тест-планів.
* **Project Managers:** для контролю відповідності розробки бізнес-вимогам.

## Single Source of Truth (SSoT) та Правила оновлення
У цьому проєкті реалізовано підхід єдиного джерела істини:
* **Вимоги:** Джерелом істини для всіх функціональних та нефункціональних вимог є виключно документ [SSD](architecture/SSD.md).
* **Архітектура:** Опис API та структурних компонентів береться лише з [SDD](architecture/SDD.md).
* **Тестування:** Test Strategy та Матриця простежуваності не дублюють текст вимог, а лише посилаються на унікальні ідентифікатори (наприклад, FR-01) з документа SSD.

**Правила оновлення:** Будь-які зміни в бізнес-логіці спочатку фіксуються в SSD, після чого оновлюються пов'язані тести. Забороняється змінювати вимоги в обхід документа SSD.

## Версійність документації (Documentation as Code)
Документація версіонується разом із вихідним кодом за допомогою Git:
1. Версія документації в гілці `main` (v1.0) завжди відповідає поточному стабільному релізу системи.
2. При додаванні нових функцій створюється нова гілка (наприклад, `feature/new-api`), де розробник паралельно оновлює код і Markdown-документи.
3. Після злиття (Merge Request) документація проходить рев'ю, і статичний сайт автоматично генерується наново, забезпечуючи 100% актуальність.
