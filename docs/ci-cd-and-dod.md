# Автоматизація та Контроль Якості Документації

## 1. Documentation CI/CD Pipeline

**Trigger (Тригери генерації):**
* Push у гілку `main` (або `develop`).
* Злиття (Merge Request) feature-гілки з `main`.

**Steps (Кроки pipeline):**
1. **Validation:** Запуск лінтера для перевірки синтаксису Markdown та валідація файлу `openapi.yaml` на відповідність стандарту OpenAPI 3.0.
2. **Build API Docs:** Генерація статичних HTML-файлів Swagger UI.
3. **Build UI Docs:** Збірка статичного сайту Storybook.
4. **Build Main Docs:** Виконання команди `mkdocs build` для збірки основного сайту документації.
5. **Publish:** Автоматичне розгортання згенерованих файлів (MkDocs, Swagger, Storybook) на платформі **GitHub Pages**.

**Versioning (Версійність):**
* Версія документації жорстко прив'язана до версії релізу продукту (наприклад, v1.0.0).
* Зміни в контракті API (breaking changes) вимагають оновлення мажорної версії документації.

## 2. Definition of Done (DoD) для документації

Цей список критеріїв гарантує, що жодна технічна зміна не потрапить у продакшен без відповідного оновлення документації.

| Тип зміни в коді | Обов'язкові дії розробника | Автоматизована перевірка (CI) |
| :--- | :--- | :--- |
| **Додано новий API ендпоінт** | Додати опис методу, параметрів та відповідей у `openapi.yaml`. | OpenAPI Validation Pipeline проходить без помилок. |
| **Змінено формат відповіді (Response)** | Оновити існуючу схему в `openapi.yaml`. | Swagger UI успішно генерується (Build). |
| **Додано/Змінено UI-компонент** | Додати або оновити відповідні сценарії (Stories) для Storybook. | Storybook Build Pipeline проходить успішно. |
| **Зміна бізнес-логіки / Архітектури** | Оновити тексти у відповідних файлах `SSD.md` або `SDD.md`. | MkDocs Build завершується без помилок збірки. |
